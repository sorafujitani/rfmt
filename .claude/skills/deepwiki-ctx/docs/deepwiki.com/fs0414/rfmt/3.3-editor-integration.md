This page documents how to integrate rfmt with code editors for automatic formatting capabilities. It covers the Ruby LSP addon architecture, editor-specific setup instructions, and configuration options for format-on-save workflows.

For programmatic formatting outside of editors, see [Ruby API](https://deepwiki.com/fs0414/rfmt/3.2-ruby-api.html). For command-line usage, see [CLI Usage](https://deepwiki.com/fs0414/rfmt/3.1-cli-usage.html).

## Overview

rfmt provides two primary integration methods for code editors:

1. **Ruby LSP Addon** - Native integration through the Ruby LSP protocol, providing format-on-save and on-demand formatting in LSP-compatible editors (VS Code, Neovim with LSP, Zed, etc.)
2. **Direct CLI Integration** - Shell command execution for editors without LSP support or for simpler setups

The Ruby LSP addon is the recommended approach as it provides seamless integration with modern editor tooling and maintains consistency with other Ruby language features.

**Sources:** [README.md277-302](https://github.com/fs0414/rfmt/blob/90c575e6/README.md#L277-L302) [docs/user\_guide.md10](https://github.com/fs0414/rfmt/blob/90c575e6/docs/user_guide.md#L10-L10)

## Architecture

### Ruby LSP Integration Architecture

The Ruby LSP addon architecture follows the LSP addon pattern where `RubyLsp::Rfmt::Addon` registers the `FormatterRunner` with the Ruby LSP server during activation. When an editor requests formatting, the LSP server dispatches to the registered formatter.

**Sources:** [lib/ruby\_lsp/rfmt/addon.rb1-21](https://github.com/fs0414/rfmt/blob/90c575e6/lib/ruby_lsp/rfmt/addon.rb#L1-L21) [lib/ruby\_lsp/rfmt/formatter\_runner.rb1-27](https://github.com/fs0414/rfmt/blob/90c575e6/lib/ruby_lsp/rfmt/formatter_runner.rb#L1-L27) [spec/ruby\_lsp/rfmt/addon\_spec.rb1-33](https://github.com/fs0414/rfmt/blob/90c575e6/spec/ruby_lsp/rfmt/addon_spec.rb#L1-L33)

### Addon Registration Flow

The addon lifecycle begins when the Ruby LSP server loads and activates registered addons. The `RubyLsp::Rfmt::Addon#activate` method instantiates a `FormatterRunner` and registers it with the global state using the identifier `'rfmt'`.

**Sources:** [lib/ruby\_lsp/rfmt/addon.rb13-15](https://github.com/fs0414/rfmt/blob/90c575e6/lib/ruby_lsp/rfmt/addon.rb#L13-L15) [spec/ruby\_lsp/rfmt/addon\_spec.rb15-25](https://github.com/fs0414/rfmt/blob/90c575e6/spec/ruby_lsp/rfmt/addon_spec.rb#L15-L25)

### Formatting Request Flow

When a formatting request arrives, the `FormatterRunner#run_formatting` method extracts the source text from the document and passes it to `Rfmt.format()`. Errors are caught and result in `nil` being returned, which signals to the LSP server that no changes should be applied.

**Sources:** [lib/ruby\_lsp/rfmt/formatter\_runner.rb11-16](https://github.com/fs0414/rfmt/blob/90c575e6/lib/ruby_lsp/rfmt/formatter_runner.rb#L11-L16) [spec/ruby\_lsp/rfmt/formatter\_runner\_spec.rb10-46](https://github.com/fs0414/rfmt/blob/90c575e6/spec/ruby_lsp/rfmt/formatter_runner_spec.rb#L10-L46)

### Core Components

| Component | File | Purpose |

### Addon Class

The `RubyLsp::Rfmt::Addon` class inherits from `RubyLsp::Addon` and implements the addon contract:

**Key methods:**

* `name` - Returns the string identifier `'rfmt'` used to select this formatter
* `activate` - Registers a new `FormatterRunner` instance with the global state
* `deactivate` - Cleanup hook (currently no-op)

**Sources:** [lib/ruby\_lsp/rfmt/addon.rb8-18](https://github.com/fs0414/rfmt/blob/90c575e6/lib/ruby_lsp/rfmt/addon.rb#L8-L18)

### FormatterRunner Class

The `FormatterRunner` implements the Ruby LSP formatter interface:

**Key methods:**

* `run_formatting(uri, document)` - Accepts a URI and document object, returns formatted source or `nil` on error
* `run_diagnostic(uri, document)` - Returns empty array (rfmt does not provide diagnostics)

**Error handling:** Any `Rfmt::Error` (including parse errors) is caught and returns `nil`, preventing the editor from receiving invalid changes.

**Sources:** [lib/ruby\_lsp/rfmt/formatter\_runner.rb7-24](https://github.com/fs0414/rfmt/blob/90c575e6/lib/ruby_lsp/rfmt/formatter_runner.rb#L7-L24) [spec/ruby\_lsp/rfmt/formatter\_runner\_spec.rb13-46](https://github.com/fs0414/rfmt/blob/90c575e6/spec/ruby_lsp/rfmt/formatter_runner_spec.rb#L13-L46)

## Neovim Setup

### Using Autocmd (Direct CLI Integration)

For Neovim users who prefer not to use LSP, rfmt can be integrated using autocmd to format files on save:

This configuration:

1. Triggers before buffer write (`BufWritePre`)
2. Matches Ruby file patterns
3. Executes `rfmt <filepath>` as a shell command
4. Reloads the buffer if formatting succeeds (exit code 0)

**Advantages:**

* Simple setup, no plugins required
* Works without Ruby LSP server
* Direct control over formatting command

**Disadvantages:**

* File must be written to disk before formatting
* No in-buffer error reporting
* Reloads entire buffer (may lose undo history)

**Sources:** [README.md279-296](https://github.com/fs0414/rfmt/blob/90c575e6/README.md#L279-L296)

| Setting | Description | Default |

## Configuration

### rfmt Configuration Discovery

The formatter respects `.rfmt.yml` configuration files using the standard discovery mechanism:

1. Current working directory
2. Parent directories (walking up the tree)
3. Home directory (`~/.rfmt.yml`)
4. Built-in defaults

For configuration file structure and options, see [Configuration](https://deepwiki.com/fs0414/rfmt/2.2-configuration.html).

### Editor-Specific Configuration

**Workspace vs User Settings:**

* Place `.rfmt.yml` in project root for team-wide consistency
* Use `~/.rfmt.yml` for personal preferences across all projects
* Workspace settings take precedence over user settings

## Troubleshooting

### Common Issues

#### Formatter Not Working

**Symptoms:** Format command does nothing or shows error

**Diagnosis:**

1. Check Ruby LSP server is running:

   * VS Code: Look for "Ruby LSP" in status bar
   * Neovim: Run `:LspInfo`
2. Verify rfmt is installed:
3. Check formatter setting:

   * VS Code: Verify `"ruby.lsp.formatter": "rfmt"` in settings
   * Neovim: Check LSP init\_options
4. Enable debug logging:

**Sources:** [docs/user\_guide.md429-456](https://github.com/fs0414/rfmt/blob/90c575e6/docs/user_guide.md#L429-L456)

#### Formatting Errors Not Shown

**Issue:** Syntax errors in code prevent formatting but no error message appears

**Explanation:** The `FormatterRunner` catches all `Rfmt::Error` exceptions and returns `nil`, which tells the LSP server to skip formatting. This is intentional to prevent applying partial/invalid changes.

**Workaround:** Run rfmt manually to see detailed error messages:

Error output will include:

* Error code (E001-E999)
* Line and column numbers
* Code snippet with error marker
* Suggested fixes

**Sources:** [lib/ruby\_lsp/rfmt/formatter\_runner.rb14-15](https://github.com/fs0414/rfmt/blob/90c575e6/lib/ruby_lsp/rfmt/formatter_runner.rb#L14-L15) [spec/ruby\_lsp/rfmt/formatter\_runner\_spec.rb25-34](https://github.com/fs0414/rfmt/blob/90c575e6/spec/ruby_lsp/rfmt/formatter_runner_spec.rb#L25-L34)

#### Format-on-Save Too Slow

**Issue:** File save is delayed by formatting operation

**Causes:**

* Very large files (>10,000 lines)
* Complex AST structures
* Slow disk I/O for cache operations

**Solutions:**

1. **Disable format-on-save for large files:**
2. **Exclude problematic files in `.rfmt.yml`:**
3. **Use async formatting (Neovim):**

**Sources:** [README.md138-143](https://github.com/fs0414/rfmt/blob/90c575e6/README.md#L138-L143) [docs/user\_guide.md393-398](https://github.com/fs0414/rfmt/blob/90c575e6/docs/user_guide.md#L393-L398)

**Issue:** Both rfmt and RuboCop trying to format simultaneously

## Best Practices

### Recommended Setup

1. **Use Ruby LSP addon** for modern editors (VS Code, Neovim with LSP, Zed)
2. **Enable format-on-save** to maintain consistent formatting automatically
3. **Commit `.rfmt.yml`** to version control for team-wide consistency
4. **Pin rfmt version** in `Gemfile` to avoid formatting differences across environments:

### Team Workflows

**For new projects:**

1. Initialize configuration: `rfmt init`
2. Format entire codebase: `rfmt .`
3. Commit changes: `git add . && git commit -m "Initial rfmt formatting"`
4. Add to CI/CD: `rfmt check .` in pipeline

**For existing projects:**

1. Format incrementally by directory
2. Create separate commits for formatting vs logic changes
3. Use `--diff` flag to review changes before applying
4. Coordinate with team to avoid merge conflicts

**Sources:** [README.md112-180](https://github.com/fs0414/rfmt/blob/90c575e6/README.md#L112-L180) [docs/user\_guide.md43-78](https://github.com/fs0414/rfmt/blob/90c575e6/docs/user_guide.md#L43-L78)
