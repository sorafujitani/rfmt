## Purpose and Scope

The **PrismBridge** serves as the Ruby-side integration layer between the Prism parser and rfmt's Rust formatting engine. It parses Ruby source code, converts Prism's native AST representation into rfmt's internal AST format, enriches nodes with metadata, and serializes the result to JSON for transmission across the FFI boundary to Rust.

This document covers the parsing and AST conversion pipeline. For information about the AST structure consumed by Rust, see [AST Representation](https://deepwiki.com/fs0414/rfmt/4.1-ast-representation.html). For details on how the formatted output is generated, see [Formatting Engine (Emitter)](https://deepwiki.com/fs0414/rfmt/4.2-formatting-engine-(emitter).html).

## Node Conversion Process

### Node Structure Mapping

The `convert_node` method transforms each Prism node into an rfmt node representation. The conversion produces a Hash with six components:

| Field | Type | Source Method | Purpose |
| --- | --- | --- | --- |
| `node_type` | String | `node_type_name()` | Identifies node variant (e.g., "class\_node") |
| `location` | Hash | `extract_location()` | Line/column/offset positions |
| `children` | Array | `extract_children()` | Child nodes (recursively converted) |
| `metadata` | Hash | `extract_metadata()` | Node-specific data (names, counts, etc.) |
| `comments` | Array | `extract_comments()` | Associated comments (empty in current implementation) |
| `formatting` | Hash | `extract_formatting()` | Formatting hints (multiline, indent, etc.) |

**Sources:** [lib/rfmt/prism\_bridge.rb86-98](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L86-L98)

## Metadata Extraction

### Metadata by Node Type

The `extract_metadata` method enriches nodes with semantic information needed for formatting. Different node types require different metadata:

| Node Type | Metadata Fields | Extraction Method |
| --- | --- | --- |
| `ClassNode` | `name`, `superclass` | `extract_class_or_module_name()`, `extract_superclass_name()` |
| `ModuleNode` | `name` | `extract_class_or_module_name()` |
| `DefNode` | `name`, `parameters_count`, `receiver` | `extract_node_name()`, `extract_parameter_count()` |
| `CallNode` | `name`, `message` | `extract_node_name()`, `extract_message_name()` |
| `StringNode` | `content` | `extract_string_content()` |
| `IntegerNode/FloatNode` | `value` | `extract_literal_value()` |
| `SymbolNode` | `value` | `extract_literal_value()` |
| `IfNode/UnlessNode` | `is_ternary` | Detect from `if_keyword_loc` |

### Handling Complex Name Extraction

Class and module names may be namespaced (e.g., `Foo::Bar::Baz`). The extractor handles three cases:

1. **ConstantReadNode**: Simple name (`User`)
2. **ConstantPathNode**: Namespaced name (`Admin::User`)
3. **Fallback**: Extract from source via `slice`

For superclasses, the extractor also handles `CallNode` cases (e.g., `ActiveRecord::Migration[8.0]`) by extracting the raw source text [lib/rfmt/prism\_node\_extractor.rb63-66](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_node_extractor.rb#L63-L66)

**Sources:** [lib/rfmt/prism\_node\_extractor.rb17-75](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_node_extractor.rb#L17-L75)

## Comment Handling

### Comment Collection

Comments are attached to the `ParseResult`, not individual nodes. PrismBridge extracts all comments and includes them in the serialized output:

The `comment_type` field distinguishes between:

* **line**: Single-line comments (`# comment`)
* **block**: Multi-line comments (`=begin...=end`)

The `position` field is initially set to `'leading'` for all comments. The Rust emitter refines this by analyzing comment positions relative to nodes [ext/rfmt/src/emitter/mod.rs86-91](https://github.com/fs0414/rfmt/blob/90c575e6/ext/rfmt/src/emitter/mod.rs#L86-L91)

**Sources:** [lib/rfmt/prism\_bridge.rb64-78](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L64-L78)

---

### Comment Preservation Strategy

The comment preservation architecture splits responsibilities:

1. **Ruby (PrismBridge)**: Collect all comments with precise location information
2. **Rust (Emitter)**:
   * Associate comments with nodes based on line numbers
   * Distinguish leading vs. trailing comments
   * Reinsert comments during formatting [ext/rfmt/src/emitter/mod.rs94-146](https://github.com/fs0414/rfmt/blob/90c575e6/ext/rfmt/src/emitter/mod.rs#L94-L146)

This separation allows the Rust side to make intelligent decisions about comment placement while formatting, using the original source positions as a guide.

**Sources:** [lib/rfmt/prism\_bridge.rb62-84](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L62-L84) [ext/rfmt/src/emitter/mod.rs85-123](https://github.com/fs0414/rfmt/blob/90c575e6/ext/rfmt/src/emitter/mod.rs#L85-L123)

## Performance Considerations

### Serialization Overhead

JSON serialization is the primary performance cost in PrismBridge. For large files, the serialization time is proportional to:

* Node count in the AST
* Depth of nesting
* Size of metadata strings

The `JSON.generate` call [lib/rfmt/prism\_bridge.rb80](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L80-L80) is implemented in C (via the `json` gem) for efficiency.

---

### Recursive Conversion

The `extract_children` method recursively converts the entire AST tree. For deeply nested code, this could theoretically cause stack overflow, but Ruby's default stack size handles typical Ruby codebases without issue.

The conversion is optimized by:

1. Using compact arrays to skip nil children [lib/rfmt/prism\_bridge.rb306](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L306-L306)
2. Failing fast on exceptions [lib/rfmt/prism\_bridge.rb300-304](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L300-L304)
3. Avoiding unnecessary object allocations

**Sources:** [lib/rfmt/prism\_bridge.rb121-307](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L121-L307)

## Limitations and Future Work

### Current Limitations

1. **Node Comments**: The `extract_comments` method on individual nodes returns empty arrays [lib/rfmt/prism\_bridge.rb370-375](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L370-L375) Comments are only available at the parse result level.
2. **Original Formatting**: The `original_formatting` field in `FormattingInfo` is always `nil` [lib/rfmt/prism\_bridge.rb386](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L386-L386) This could store original source text for specific nodes.
3. **Node Type Coverage**: While 80+ node types are supported, some rare Ruby constructs may fall back to `Unknown` variants.

---

### Potential Enhancements

Future improvements could include:

* **Streaming JSON**: For very large files, emit JSON incrementally rather than building the entire structure in memory
* **Direct FFI**: Pass Prism AST nodes directly to Rust without JSON intermediary (requires deeper Magnus integration)
* **Caching**: Cache parsed ASTs for unchanged files to skip redundant parsing
* **Parallel Parsing**: Parse multiple files concurrently for directory formatting operations

**Sources:** Architecture analysis based on [lib/rfmt/prism\_bridge.rb1-390](https://github.com/fs0414/rfmt/blob/90c575e6/lib/rfmt/prism_bridge.rb#L1-L390)
